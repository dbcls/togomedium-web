## 概要

- このガイドは、`AGENTS.md` や `docs/` 配下のドキュメントを編集する時に参照するためのものです。
- 各種ドキュメントは AI だけでなく人間が読んでも理解しやすい文章・ボリュームであるべきです。
- コードの内容をそのまま転記するのではなく、コードだけでは分かりづらい前提、判断基準、歴史的経緯を補うことを目的にしてください。

## 基本方針

- まずコード、既存ドキュメント、関連する `AGENTS.md` を確認してから書き始めてください。
- コード構成やファイル配置をそのまま並べるだけの説明は避け、読み手が判断に迷う箇所を優先して補ってください。
- 実装の理由、責務分担、例外運用、長期的な方針のように、コードを見ただけでは読み取りづらい情報を優先してください。
- 明らかにコードから分かる内容は最小限にとどめ、必要なら背景や判断理由に言い換えてください。

## 書き方の考え方

- ルート `AGENTS.md` にはモノリポ全体で長く使う運用ルールを残してください。
- 各パッケージの `AGENTS.md` には、その配下で迷いやすい前提と関連ガイドへの導線を残してください。
- より詳しい実装指針や背景説明は `docs/` 配下に分離してください。
- 一時的な検討メモや短期間で陳腐化する内容は、恒久ドキュメントに直接書き込まないでください。

## 情報が足りない時

- コードや既存ドキュメントを読んでも判断できない点が重要な場合は、必要に応じてユーザーへ背景情報や運用方針を確認してください。
- 特に、過去の経緯、例外扱い、将来方針はコードだけでは確定できないことがあるため、推測で断定しないでください。
- すぐに確定できない内容は、必要に応じてメモや TODO として分け、確定ルールと混ぜないでください。



## 日本語特有の記法について

### 半角スペース
- 日本語と英数字・英語表現の間には、原則として半角スペースを入れない。
- ただし、英語表現の内部に含まれる語と語の区切りは、元の表記どおり半角スペースを保持する。
- 数字と日本語の助数詞・単位の間にも、原則として半角スペースを入れない。
- インラインコードを文中で使う場合は、バッククォートの前後に半角スペースを入れる。
- インラインコード内では、コードや記法そのものに含まれる半角スペースをそのまま保持する。
- 例
    - NG `TypeScript について`
    - OK `TypeScriptについて`
    - NG `2 種類`
    - OK `2種類`
    - OK `npm scriptを実行する`
    - OK `CSS Gridの仕様`
    - OK `` `インラインコード` を使う ``
    - OK `` `npm run dev` を実行する ``

### 文末表現
- 箇条書きに限らず、同一段落内や連続する項目内で、同じ文末表現の繰り返しが続きすぎないように調整する。
- 特に `している` `である` `とする` などが機械的に並ぶ場合は、意味を変えない範囲で、言い換え・文の組み替え・省略などを用いて単調さを避ける。
- 文章を書き足した場合は、その行だけで判断せず、前後の文や隣接する箇条書きも確認したうえで文末の重複を調整する。
- 同種の処理や性質を並列に列挙する箇所では、文末をそろえたほうが意味が明確になる場合があるため、無理に変化を付けない。
- 文書一覧や補足一覧のように、対象や役割を短く示す箇所では、体言止めの使用も検討する。
- 見出しや文脈から共有できている情報は、各箇条書きの動詞で繰り返さない。
    - 例えば、一覧や構成説明では、`配置する` `置く` のような自明な説明よりも、その項目が何を担うかを優先する。
    - 見出しだけで役割が明確な場合は、各項目でも同じ役割語を繰り返さず、補足すべき差分だけを書く。
- 文末表現は、統一感と読み進めたときのリズムの両方を考慮して調整する。特に、意味上の並列性が重要な箇所では統一を優先する。
