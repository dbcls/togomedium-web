## 概要

- このガイドでは、TogoMedium Webリポジトリ内のドキュメント配置、記載対象、判断基準の扱いを定める。
- `AGENTS.md` や `docs/` 配下のドキュメントを新規作成、整理、更新するときに参照する。
- ドキュメントは、AIだけでなく人間も読める文章量と粒度にする。
- コードだけでは読み取りづらい前提、判断基準、例外、経緯を補う。

## 編集前に確認すること

- 対象ディレクトリと関係するパッケージを確認する。
- 関連するコード、既存ドキュメント、`AGENTS.md` を確認する。
- 既存の未コミット変更がある場合は、現在の内容を前提にして編集する。
- 変更が別のドキュメントや作業ルールにも影響する場合は、必要な範囲だけ確認する。

## 配置と役割

- ルート `AGENTS.md` には、モノリポ全体で長く使う運用ルール、判断基準、共通の前提を置く。
- 各ディレクトリの `AGENTS.md` には、その配下で作業するときに迷いやすい前提、例外、関連ガイドへの導線を置く。
- 詳しい実装指針、背景説明、長い手順は `docs/` 配下に分離する。
- プロジェクト内ドキュメントを参照するときは、Markdownリンクではなく `docs/...` のようなパス表記を使う。

## 書く内容

- 読み手が判断に迷う前提を優先して書く。
- 実装の理由、責務分担、例外運用、長期的な方針を明示する。
- コードから明らかに分かる内容は最小限にとどめる。
- コード構成やファイル配置を示す場合は、一覧そのものではなく、作業判断に必要な意味を補う。

## 書かない内容

- 一時的なタスクメモ、作業ログ、短期間で陳腐化する議論は恒久ドキュメントに混ぜない。
- 過去の会話ログをそのまま転載しない。
- 未確定の運用方針を確定ルールのように書かない。
- 実装の都合に合わせて、既存の判断基準を説明なく変更しない。

## 情報が足りないとき

- コードや既存ドキュメントを読んでも重要な判断ができない場合は、ユーザーに背景情報や運用方針を確認する。
- 過去の経緯、例外扱い、将来方針は、推測で断定しない。
- すぐに確定できない内容は、メモや TODOとして分け、確定ルールと混ぜない。

## 完了前に確認すること

- 変更内容が、対象ドキュメントの役割から外れていないことを確認する。
- 関連する `AGENTS.md` や `docs/` への参照が、現在の配置と一致していることを確認する。
- 一時的な記録と長く使うルールが混ざっていないことを確認する。
- ドキュメントのみの変更であれば、format / lint / testは原則として不要とする。
